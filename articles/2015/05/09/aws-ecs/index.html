Launch wordpress in aws ec2 container service

May 9, 2015

# docker # AWS ECS # wordpress

Amazon Web Service (AWS) have launched a new service EC2 container service (ECS) in several regions (without singapo region currently). In short, It lets you manage docker containers running in EC2 instances.

If you are using "Amazon ECS sample" task definition to launch ECS service during the "get started" tutorail, AWS will use cloudformation template to launch all components needed.

Like AWS EC2 service, you are responsible to prepare the VPC(s), Security Group(s), IAM Role(s) and optional ELB(s) when you are planning to create a custom cluster.

Definitions

There are several terms used by ECS:

  • "Task"

It is used to define container settings. One task per one EC2 instance.

For instance, in order to run wordpress, I need to define three container settings in a task named "prod-wordpress", one docker run nginx, another one run php5-fpm and the last one run a data volume. AWS ECS will start three docker containers in one EC2 instance after I run the task.

  • "Service"

It is used to define which task to run, where to run it (in which ECS cluster) and how many tasks run. ELB can be used to load balance a sepecific port.

  • "Cluster"

It is an EC2 instance pool. The container agent installed in the EC2 instance will register the EC2 instance to the cluster defined in it's config.

For details, please refers to "Getting Started with Amazon ECS"

Customization

In order to launch my own "wordpress" project, I choose to create the infrastructure from scrach. Following as a summary (production and stagging environment in the same VPC), ensure high availability of wordpress by defining subnets accross multiple AZs:

vpc:
  "wordpress":
    cidr: 10.10.0.0/16
subnets:
  "wordpress-web1":
    cidr: 10.10.1.0/24
    az: a
  "wordpress-web2":
    cidr: 10.10.2.0/24
    az: b
  "wordpress-db1":
    cidr: 10.10.10.0/24
    az: a
  "wordpress-db2":
    cidr: 10.10.11.0/24
    az: b
security_groups:
  "prod-wordpress-elb":
    - "tcp:80:cidr:0.0.0.0/0"
    - "tcp:443:cidr:0.0.0.0/0"
  "prod-wordpress-web":
    - "tcp:80:ref:prod-wordpress-elb"
  "prod-wordpress-db":
    - "tcp:3306:ref:prod-wordpress-web"
iam_roles:
  "prod-wordpress-web":
    "s3-upload":
      action:
        - "s3:*"
      resource:
        - "arn:aws:s3:::prod-wordpress/upload" 
        - "arn:aws:s3:::prod-wordpress/upload/*"

Then create the following:

1) An EC2 auto scaling group using subnets "wordpress-web1" and "wordpress-web2", security group "prod-wordpress-web" and IAM role "prod-wordpress-web". Those EC2 instances will auto join EC2 cluster named "default".

2) A RDS instance using subnets "wordpress-db1" and "wordpress-db2", security group "prod-wordpress-db".

3) An ELB using subnets "wordpress-web1" and "wordpress-web2", security group "prod-wordpress-elb"
